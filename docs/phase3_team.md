## Git チーム開発の基本フロー

```
main ─────────────────────────────────────────►
       │                              ▲
       │ branch                       │ merge
       ▼                              │
feature/xxx ──► commit ──► push ──► PR ──► review ──┘
```

### 各ステップの役割

| ステップ | やること | なぜ必要か |
|---------|---------|----------|
| Branch | mainから新しいブランチを作る | mainを壊さずに作業するため |
| Commit | 変更を記録する | いつ・何を変えたかの履歴を残すため |
| Push | GitHubに送る | チームメンバーが見られるようにするため |
| PR | プルリクエストを作る | 変更をチームに提案するため |
| Review | PRを確認する | 間違いを防ぎ、学び合うため |
| Merge | mainに統合する | 変更をチームの成果物に反映するため |

## チーム開発の心得

### やるべきこと

1. **こまめにコミット** — 大きな変更を一度にしない
2. **わかりやすいコミットメッセージ** — 何を変えたか明確に
3. **PRは小さく** — レビューしやすいサイズで
4. **コンフリクトは相談** — 勝手に解決しない
5. **作業前にpull** — 最新のmainを取得してから作業開始

### やってはいけないこと

1. **mainに直接プッシュ** — 必ずブランチを切る
2. **レビューなしでマージ** — 必ず誰かに見てもらう
3. **他人の変更を勝手に消す** — 必ず相談する
4. **テストせずにプッシュ** — 動作確認してから
